The "Blank Screen" Pathological FailureLocation: Likely client/App.jsx and your Context Providers.The Diagnostic: When you switch companies, your companyId changes in localStorage, but your React state does not "flush" instantly.The Bug: You have a component (likely a Dashboard Widget or a Sales Table) that tries to map over an array of data. When the switch happens, the component re-renders before the new data has arrived from the backend. It tries to access a property (like lead.status) on a null or undefined object.The Critical Error: Because you lack Global Error Boundaries, one tiny "undefined" error at the bottom of the component tree kills the entire DOM, resulting in the "White Screen of Death."Deep Fix: You must wrap the main dashboard area in a <ErrorBoundary fallback={<ErrorMessage />}> and add a key={activeCompanyId} to your main content wrapper. This forces a complete "unmount/remount" of the UI, ensuring no data from Company A "ghosts" into Company B.2. The "Security by String" LiabilityLocation: functions/auditDataIsolation.ts and functions/checkSubscriptionLimits.ts.The Diagnostic: I see logic that checks for specific email strings (e.g., if (user.email === 'kevin@...')) to grant admin rights or bypass restrictions.The Audit: This is the #1 way SaaS platforms get hacked or break during updates. Hard-coding identity logic inside functional files creates a "spaghetti security" model. If you change your email, or if you want to add a partner, you have to manually edit 10+ files.The Deep Fix: You need to move this to the Database Level. Your Users table should have a platform_role column. Your code should only check if (user.role === 'SUPER_ADMIN'). Never use an email address as a permission key.3. Database "Cleanup" vs. "Integrity"Location: The 12+ cleanup files (cleanupDuplicateCompanies.ts, deleteDuplicateLeads.ts, etc.).The Diagnostic: The presence of these files proves that your Intake Logic (the code that saves leads and companies) is failing to validate data at the front door.The Risk: As you scale to hundreds of users, these scripts will become dangerously slow.
